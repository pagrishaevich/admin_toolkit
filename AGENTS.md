AGENTS.md

Поведенческие и операционные правила для больших долгоживущих проектов.

Эти правила оптимизированы для:

* OpenCode
* Codex
* GPT-5.5
* больших репозиториев
* многосессионной разработки
* минимального расхода контекста
* сохранения состояния проекта между сессиями

⸻

1. Основные инженерные принципы

Думай перед тем как писать код

Не делай предположений.
Не выбирай неоднозначные трактовки молча.
Показывай компромиссы заранее.

Перед реализацией:

* явно озвучивай предположения при неопределенности
* задавай уточняющие вопросы если требования неясны
* предпочитай максимально простую реализацию
* сопротивляйся ненужной сложности
* кратко объясняй важные компромиссы

Если что-то неясно:

* остановись
* укажи что именно непонятно
* задай точечные вопросы

⸻

Сначала простота

Пиши минимально необходимый код.

Правила:

* никаких спекулятивных фич
* никаких преждевременных абстракций
* никаких одноразовых абстракций
* никакой лишней конфигурируемости
* никакого расширения архитектуры без необходимости
* никакого оверинжиниринга

Всегда задавай вопрос:

«Посчитает ли senior-инженер это неоправданно сложным?»

Если да:

* упрощай
* уменьшай количество слоев
* уменьшай количество абстракций
* сокращай объем кода

⸻

Только точечные изменения

Изменяй только то, что действительно требуется.

При редактировании кода:

* не рефактори посторонний код
* не форматируй несвязанные участки
* не переименовывай несвязанные сущности
* соблюдай стиль существующего проекта
* избегай массовых переписываний

Разрешенная очистка:

* импорты, ставшие неиспользуемыми ИЗ-ЗА ТВОИХ изменений
* переменные/функции, ставшие неиспользуемыми ИЗ-ЗА ТВОИХ изменений
* прямое дублирование, появившееся ИЗ-ЗА ТВОИХ изменений

Если обнаружены сторонние проблемы:

* упомяни их
* не исправляй без запроса

Каждая измененная строка должна напрямую относиться к задаче.

⸻

Ограничения на рефакторинг

Рефакторинг запрещен если он:

* не нужен для выполнения задачи
* не исправляет баг
* не устраняет дублирование, возникшее из-за текущих изменений
* не был явно запрошен

Не перестраивай архитектуру во время обычной разработки фич.

⸻

2. Работа через цели

Преобразовывай расплывчатые задачи в проверяемые цели.

Примеры:

* «Исправить баг» → воспроизвести → исправить → проверить
* «Добавить валидацию» → создать невалидные кейсы → добиться прохождения тестов
* «Рефакторинг модуля» → убедиться что поведение не изменилось

Для сложных задач используй:

1. [шаг] → проверка: [что проверить]
2. [шаг] → проверка: [что проверить]
3. [шаг] → проверка: [что проверить]

Предпочитай измеримые критерии успеха.

Избегай:

* «сделать лучше»
* «улучшить архитектуру»
* «почистить всё»

⸻

3. Управление контекстом и токенами

Контекст должен оставаться компактным и эффективным.

Правила работы с контекстом

* сначала изучай структуру проекта
* читай только файлы, относящиеся к задаче
* никогда не загружай весь репозиторий в контекст
* избегай dependency-директорий и сгенерированных файлов
* избегай огромных логов и build-артефактов
* сначала используй поиск, потом открывай файлы
* большие файлы читай только нужными секциями
* суммируй результаты вместо вставки огромных фрагментов

Дисциплина токенов

* не повторяй большие блоки кода
* не повторяй старые summary
* сжимай повторяющуюся историю
* держи активный контекст сфокусированным на текущей задаче
* используй краткую техническую коммуникацию
* предпочитай diff и точечные фрагменты вместо целых файлов

Работа с логами

Для логов и вывода команд:

* анализируй только релевантную часть
* обычно достаточно последних 100–200 строк
* расширяй область анализа только при необходимости

⸻

4. Сначала существующая архитектура

Не придумывай новую архитектуру без необходимости.

Правила:

* переиспользуй существующие паттерны
* переиспользуй существующие утилиты
* переиспользуй существующую инфраструктуру
* предпочитай консистентность идеализму
* расширяй существующие модули вместо создания новых систем

Избегай:

* внедрения фреймворков без необходимости
* создания абстракций «на будущее»
* замены рабочих систем в рамках несвязанных задач

⸻

5. Система постоянной памяти проекта

Основное правило

Постоянная память проекта хранится в .memory/.

.memory/ — основной источник восстановления состояния проекта между сессиями.

Там хранятся:

* архитектура
* текущий прогресс
* активные задачи
* нерешенные баги
* принятые решения
* важные детали реализации
* последние изменения

⸻

6. Стартовый workflow

Перед началом ЛЮБОЙ задачи:

1. Прочитать все файлы внутри .memory/
2. Восстановить понимание текущего состояния проекта
3. Определить активные задачи и нерешенные проблемы
4. Изучить только код, относящийся к текущей задаче
5. Продолжить работу с последнего известного состояния

Никогда не начинай реализацию без восстановления контекста проекта.

⸻

7. Структура memory-файлов

.memory/current_state.md

Содержит:

* текущую архитектуру
* активные модули
* важные файлы
* статус интеграций
* последнее стабильное состояние реализации

Правила:

* держать кратким
* держать актуальным
* избегать исторического мусора
* желательно не превышать ~300–500 строк

Исторические детали должны храниться в session logs.

⸻

.memory/tasks.md

Содержит:

* активные TODO
* завершенные задачи
* приоритеты
* ожидающие исправления
* известный технический долг

Правила:

* всегда обновлять статусы
* удалять устаревшие TODO
* сохранять нерешенные проблемы
* задачи должны быть краткими и конкретными

⸻

.memory/decisions.md

Содержит:

* архитектурные решения
* выбор фреймворков
* API-контракты
* решения по безопасности
* важные компромиссы

Правила:

* дописывать историю вместо переписывания
* фиксировать ПОЧЕМУ было принято решение
* фиксировать последствия если это важно

НЕ включать:

* временные детали реализации
* мелкие изменения кода
* шумные session notes

⸻

.memory/session_logs/YYYY-MM-DD.md

Содержит краткие хронологические summary сессий.

Каждая запись должна включать:

* что изменилось
* почему это изменилось
* измененные файлы
* нерешенные проблемы
* рекомендуемые следующие шаги

Правила:

* записи должны добавляться хронологически
* писать кратко, но информативно
* сжимать повторяющуюся информацию
* сохранять важные детали реализации

⸻

8. Обязательный workflow после задачи

После завершения важной работы:

1. Обновить .memory/current_state.md
2. Обновить .memory/tasks.md
3. Обновить .memory/decisions.md если изменилась архитектура
4. Добавить summary в .memory/session_logs/YYYY-MM-DD.md
5. Проверить консистентность memory-файлов

Обновлять memory только если:

* изменилась архитектура
* изменился статус задачи
* найден важный баг
* завершена фича
* добавлена зависимость
* существенно изменилось направление реализации

Избегай лишнего шума в memory.

⸻

9. Правила сжатия контекста

Если контекст проекта становится слишком большим:

* суммируй старые сессии
* сжимай повторяющуюся историю
* сохраняй архитектуру
* сохраняй активные задачи
* сохраняй нерешенные баги
* сохраняй критичные детали реализации
* сохраняй долгосрочные решения

Никогда не теряй:

* архитектурную мотивацию
* нерешенные проблемы
* активные задачи
* интеграционные предположения
* API-контракты

⸻

10. Правила документации

При изменении кода:

* упоминай затронутые файлы в memory
* документируй неочевидное поведение
* документируй миграции
* документируй breaking changes
* документируй новые зависимости
* документируй важные компромиссы

Документация должна оставаться краткой и технической.

⸻

11. Антипаттерны

Избегай:

* оверинжиниринга
* спекулятивных абстракций
* массового рефакторинга
* ненужных переписываний
* переполнения контекста
* скрытых предположений
* тихих архитектурных изменений
* загрузки огромных файлов в контекст
* замены стабильных систем без причины

⸻

12. Критерии успеха

Эти правила работают правильно если:

* diff остаются небольшими и точечными
* контекст остается управляемым
* состояние проекта переживает сброс сессии
* требуется меньше переписываний
* архитектура остается стабильной
* задачи восстанавливаются между сессиями
* memory-файлы остаются компактными и полезными
* реализация остается согласованной с существующим кодом
