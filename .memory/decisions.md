# Решения

## 2026-05-14: настройка chrony должна работать без неподключённых drop-in

Решение: `scripts/time.sh` пишет drop-in в `/etc/chrony.conf.d` только если основной `chrony.conf` явно подключает этот каталог через `confdir` или `include`. Иначе скрипт редактирует основной конфиг.

Почему: наличие каталога `/etc/chrony.conf.d` само по себе не гарантирует, что chrony читает файлы из него.

## 2026-05-14: вернуть реалистичный таймаут Kaspersky setup

Решение: дефолт `KASPERSKY_SETUP_TIMEOUT` увеличен до 300 секунд, `KASPERSKY_SETUP_KILL_AFTER` — до 10 секунд.

Почему: 5 секунд недостаточно для стабильной тихой первичной настройки `kesl-setup.pl`; процесс мог завершаться принудительно до фактической настройки KESL.

## 2026-05-14: использовать short hostname для доменных операций

Решение: для доменной валидации и ввода в домен используется `hostname -s`, а не полный вывод `hostname`.

Почему: на тестовом RED OS-хосте `hostname` возвращает FQDN вида `GUB-ADM-TEST.yg.loc`, а доменный join требует короткое имя NetBIOS-формата до 15 символов.

## 2026-05-14: time должен быть идемпотентным по makestep

Решение: при редактировании основного `chrony.conf` удаляются старые строки `server`, `pool` и `makestep`, затем добавляется актуальная пара `server`/`makestep`.

Почему: повторные запуски `time` накапливали строки `makestep 1.0 3`.

## 2026-05-15: chrony использует два сервера времени

Решение: основной сервер времени остаётся `time.yanao.ru iburst`, дополнительный сервер задаётся через `NTP_EXTRA_SERVERS` и по умолчанию равен `10.82.200.1`.

Почему: на целевых хостах в `/etc/chrony.conf` должны присутствовать обе строки: `server time.yanao.ru iburst` и `server 10.82.200.1`.

## 2026-05-15: Kaspersky GUI устанавливается по умолчанию

Решение: `KASPERSKY_INSTALL_GUI` по умолчанию включён (`1`), а `postcheck` проверяет RPM `kesl-gui`.

Почему: при стандартной установке должен ставиться пакет `kesl-gui-12.2.0-2412.x86_64.rpm`.

## 2026-05-15: разные права для CIFS-шар inv и distr

Решение: `/mnt/inv` монтируется с `rw,noperm,file_mode=0777,dir_mode=0777`, а `/mnt/distr` монтируется с `ro`.

Почему: `inv` — открытая шара для записи отчётов и инвентаризации, `distr` должен оставаться только для чтения.

## 2026-05-15: автоматическое принятие лицензии ViPNet

Решение: установка ViPNet использует `yes YES | dnf install -y ...`, а код возвращает статус именно `dnf`.

Почему: установщик ViPNet может запрашивать `YES` во время установки RPM, и фиксированного количества строк `YES` недостаточно для надёжной тихой установки.

## 2026-05-25: Ассистент входит в штатный шаг software

Решение: новый модуль `scripts/assistant.sh` выполняется после ViPNet, устанавливает `assistant-fstek-*.x86_64.rpm` из `/mnt/distr/linux/bootstrap/assistant` и дописывает отсутствующие строки из `redos_hosts.txt` в конец `/etc/hosts`.

Почему: Ассистент нужен в стандартной установке рабочего места вместе с Kaspersky, КриптоПро и ViPNet, а его узлы должны быть доступны сразу после установки без ручной правки hosts.

Последствия:

- Повторный запуск не дублирует строки в `/etc/hosts`.
- `postcheck` проверяет RPM `assistant-fstek` и наличие строк из `redos_hosts.txt`.

## 2026-05-25: после Ассистента настраивается рабочее окружение KDE

Решение: модуль `scripts/assistant.sh` после установки Ассистента устанавливает группу `@KDE` с `--allowerasing`, пакет `sddm`, отключает `gdm`, включает `sddm` и устанавливает `redos-wintheme-switcher`.

Почему: стандартно настроенный компьютер должен получать KDE и Windows-like оформление в том же автоматическом сценарии.

Последствия:

- Управление рабочим окружением включено параметром `ASSISTANT_DESKTOP_ENABLED=1`.
- `postcheck` проверяет пакет темы, наличие и включение `sddm`, а также что `gdm` не включён.

## 2026-05-25: CI проверяет shell-скрипты в стиле проекта

Решение: `scripts/validate.sh` запускает `shellcheck -x -P "$DIR"` для разрешения локальных `source` и `shfmt -d -i 2` для проверки форматирования с двухпробельным отступом.

Почему: прежний CI не видел подключаемые файлы и требовал табуляцию `shfmt` по умолчанию, хотя существующие скрипты используют два пробела; из-за этого workflow был красным до проверки поведения.

Последствия:

- Shell-скрипты механически нормализованы форматом `shfmt -i 2`.
- Ошибки `shellcheck`, скрытые прежней конфигурацией, исправлены точечно.
